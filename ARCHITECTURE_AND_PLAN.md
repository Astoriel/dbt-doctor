# Проект: dbt-sage (MCP Server for Data Engineering & dbt)

## 🏗 Подробнейшая архитектура проекта

**dbt-sage** (или `mcp-dbt-copilot`) — это сервер для локальной или облачной работы, реализующий протокол MCP (Model Context Protocol). Его главная задача — предоставить любой современной LLM (будь то Claude Desktop, Cursor, Windsurf или кастомный скрипт) легальный, структурированный доступ (Tools) к вашему Data Warehouse и кодовой базе dbt.

### Как это должно работать? (User Flow)
1. Пользователь разрабатывает SQL-модель в dbt (например, `fct_users.sql`).
2. В клиенте (например, Claude Desktop) он пишет: *"Проанализируй модель `fct_users.sql`, посмотри реальные данные в базе, исправь ошибки в SQL и напиши к ней исчерпывающий `schema.yml` с бизнес-тестами"*.
3. LLM под капотом обращается к инструментам (Tools) **твоего MCP сервера**:
   - Вызывает `get_dbt_models`, чтобы найти путь к SQL-файлу.
   - Вызывает `get_model_code`, чтобы прочитать SQL.
   - Вызывает `get_upstream_lineage`, чтобы посмотреть, откуда берутся базовые данные (парсинг `manifest.json`).
   - Вызывает `profile_dbt_model`, который отправляет SQL-запрос в реальную базу данных (Postgres/Snowflake), считает количество `NULL` значений, уникальных ID и достает 10 примеров строк.
   - Вызывает `write_dbt_yaml`, который вставляет сгенерированные описания колонок и тесты (например, `unique`, `not_null`, `accepted_values`) прямо в кодовую базу пользователя.
4. Вуаля! Модель задокументирована и покрыта тестами на основе статистики реальной базы данных.

### 🛠 Стек технологий (С помощью чего реализован)
1. **Язык (Language):** Python Python 3.10+ (основной язык в Data Engineering).
2. **MCP SDK:** `mcp` (официальный пакет от Anthropic). Будем использовать `fastmcp` для быстрого роутинга инструментов (Tools) аналогично FastAPI.
3. **dbt Integration:** Библиотека `dbt-core`. Вместо вызова команд в терминале сервер будет программно парсить её DWH-артефакты (`target/manifest.json`, `target/catalog.json`).
4. **Data Warehouse Driver:** `psycopg` (для Postgres) или универсальный `SQLAlchemy` (для прямых запросов профилирования к базе). Сервер будет находить креды в `~/.dbt/profiles.yml`.
5. **Smart YAML Editing:** `ruamel.yaml` — это критично. Стандартный `PyYAML` удаляет комментарии людей и форматирование при перезаписи файлов. `ruamel.yaml` умеет делать in-place update словаря, сохраняя ручные комментарии, пустые строки и порядок ключей.
6. **Transport:** `stdio` (Standard Input/Output). Claude Desktop будет общаться с этим сервером не по сети (что вызвало бы проблемы с безопасностью доступов к БД), а локально через `stdin/stdout` потоки операционной системы.

## 📅 Подробный To-Do План имплементации (Roadmap)

Проект разбит на 4 спринта. Я создам новую рабочую папку и положу этот файл туда.

### 🟢 Phase 1: Базовый скелет и Чтение Контекста (Read-Only)
Цель: Сделать так, чтобы LLM могла "видеть" проект dbt и его манифесты.

* [ ] Инициализировать проект (папка `Проект 4 dbt-mcp-server`), создать `pyproject.toml` (с помощью пакета `uv` или `poetry`).
* [ ] Установить `mcp` SDK и создать `FastMCP("dbt-sage")` инстанс сервера в `main.py`.
* [ ] Написать логику `dbt_parser.py`: функция, которая находит и парсит `target/manifest.json`.
* [ ] Реализовать инструмент (MCP Tool) `list_models`: Возвращает LLM список всех моделей / view (с их алиасами, путями и описаниями) в проекте.
* [ ] Реализовать инструмент (MCP Tool) `get_model_details`: По имени модели возвращает её сырой код (`.sql`), скомпилированный код (`target/compiled/`) и текущий фрагмент `schema.yml`.
* [ ] Реализовать инструмент `get_lineage`: Выдает граф зависимостей `depends_on`.

### 🟡 Phase 2: Интеграция с Базой Данных (Data Profiler)
Цель: Дать LLM возможность делать запросы напрямую к реальной БД (чтобы генерировать релевантные тесты).

* [ ] Написать парсер файла профилей `profiles.yml` (искать в корне проекта или в `~/.dbt/`), чтобы автоматически получать БД credential (host, user, password, dbname).
* [ ] Написать обертку базы (Base Connection Manager) для Postgres.
* [ ] Реализовать инструмент `execute_sql`: Позволяет LLM выполнять произвольные безопасные (read-only) SQL скрипты и возвращать ответ в JSON формате. LLM сможет сама тестировать свои гипотезы насчет данных.
* [ ] Реализовать инструмент `profile_model`: Инструмент сам генерирует SQL запрос: считает `MIN()`, `MAX()`, `COUNT(DISTINCT)`, `%_NULLS` для каждой колонки текущей модели. Возвращает статистику для LLM.

### 🟠 Phase 3: Мутация Кодовых Баз (YAML Writer)
Цель: Наделить LLM "руками", чтобы она сама писала код в проект пользователя.

* [ ] Создать `yaml_manager.py` на основе `ruamel.yaml`.
* [ ] Написать логику поиска `schema.yml` для модели (он часто лежит в одной папке с `name.sql`, либо в папке `/models/` как общий файл).
* [ ] Реализовать инструмент `update_model_yaml`: Принимает JSON-payload (описание модели, описание колонок, список тестов), аккуратно парсит текущий `.yml` файл, мержит изменения и перезаписывает его in-place (не ломая остальной конфиг).
* [ ] Обработка исключений: возможность создать новый `_models.yml`, если его не было.
